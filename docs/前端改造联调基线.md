## 2. 当前已统一的产品流程

## 2.1 主图主链路

主图正式链路固定为：

`upload -> analyze -> platform -> generate -> strategy -> result -> payment -> hd-result`

关键说明：

- `generate` 后面直接进入 `strategy`
- `copywriting` 不属于主图主链路
- `result` 是主图结果页
- `payment` 是主图高清支付页
- `hd-result` 是主图高清无水印结果页

## 2.2 详情图链路

详情图链路固定为：

`hd-result -> copywriting -> detail-confirm -> detail-result`

关键说明：

- `copywriting` 在这里是详情图文案确认页
- `detail-confirm` 是详情图方案确认页
- `detail-result` 是详情图结果页

## 2.3 登录入口的最终位置

登录/注册入口只允许放在两个页面：

- `hd-result`
- `detail-result`

并且登录是：

- 可选动作
- 不是前置门禁
- 不是支付前门禁
- 不是主图结果页门禁

它的职责是：

- 让用户把当前真实 session 认领到账户
- 让主图/高清图/详情图资产进入该账户 `History`

## 3. 当前后端已确认的关键契约

## 3.1 历史列表

`GET /api/v2/account/assets`

返回结构不是数组根节点，而是：

```json
{
  "items": [],
  "total": 0,
  "page": 1,
  "page_size": 20
}
```

支持筛选参数：

- `q`
- `platform_id`
- `image_type`
- `style_tag`
- `brand_name`
- `page`
- `page_size`

注意：

- 该接口不返回 session status
- History 的完成/处理中/失败/未完成，需要前端补拉 `GET /sessions/{id}`

## 3.2 Session 快照

`GET /api/v2/sessions/{session_id}`

当前前端重点使用字段：

- `session_id`
- `status`
- `current_step`
- `selected_platform_ids`
- `active_platform_id`
- `analysis_snapshot`
- `parameter_snapshot`
- `confirmed_copy`
- `strategy_preview`
- `generation_round`
- `latest_result_version`
- `detail_generation_round`
- `detail_latest_result_version`

## 3.3 参数相关接口

参数附件：

- `POST /api/v2/sessions/{id}/parameter-attachments`
- `GET /api/v2/sessions/{id}/parameter-attachments`
- `DELETE /api/v2/sessions/{id}/parameter-attachments/{attachment_id}`

参数提取：

- `POST /api/v2/sessions/{id}/parameters/extract`
- `GET /api/v2/jobs/{job_id}`
- `GET /api/v2/sessions/{id}/parameters`

参数保存：

- `PUT /api/v2/sessions/{id}/parameters`

注意：

- 参数提取是异步 job，不是同步返回
- 上传时需要 `multipart/form-data`
- 附件上传需要 `display_order`

## 3.4 主图策略与结果

主图策略预览：

- `POST /api/v2/sessions/{id}/strategy/preview`

主图整组生成：

- `POST /api/v2/sessions/{id}/generations`

主图结果：

- `GET /api/v2/sessions/{id}/results?version=`

返回结构中前端需要保留：

- `latest_result_version`
- `requested_version`
- `available_versions`
- `version_summaries`

## 3.5 详情图策略与结果

详情图策略预览：

- `POST /api/v2/sessions/{id}/detail-pages/strategy/preview`

详情图整组生成：

- `POST /api/v2/sessions/{id}/detail-pages/generations`

详情图结果：

- `GET /api/v2/sessions/{id}/detail-pages/results?version=`

详情图下载：

- `GET /api/v2/sessions/{id}/detail-pages/download`

## 4. 当前前端已落地的改造方向

## 4.1 API 适配层

当前前端已在 `client/src/lib/api.ts` 收口这些问题：

- `/account/assets` 按 `{ items, total, page, page_size }` 解包
- `/sessions/{id}/images` 按 `{ images }` 解包
- `/sessions/{id}/results` 标准化为统一版本结果结构
- `downloadResults(sessionId, version?)` 支持按当前版本下载
- 参数提取改为 `trigger job -> poll job -> reload snapshot`
- 钱包、价格、附件、参考图统一做数组兼容处理

## 4.2 结果页版本管理

当前前端结果页已按后端真实结构恢复：

- 默认显示最新版本
- 多版本切换
- 单图重生 / 整体修改后切回最新版本
- 当前版本写入 `sessionStorage.current_result_version`
- 支付 / 高清链路绑定当前查看版本

## 4.3 History

History 当前按“服务端筛选 + 前端状态补全”实现：

- 支持平台、图片类型、风格、品牌、状态、关键词筛选
- 状态通过 session snapshot 二次归类
- 空白 session 不进入 History
- 已完成详情支持版本查看和下载
- 未完成/失败支持查看和继续流程

## 4.4 结果页、支付页、高清页

当前已补齐：

- `result`
- `payment`
- `hd-result`

并已接上：

- 主图版本上下文
- 选图上下文
- 高清页大图展示
- 单张下载
- 当前版本整包下载

需要特别说明：

- 当前后端没有独立的“主图高清无水印订单/交付”接口
- 所以这条链路前端只能先按模板流程组织，真实后端订单能力仍待补

## 4.5 详情图

当前详情图链路已接到真实后端接口：

- `detail-confirm` 会取真实 `detail-pages/strategy/preview`
- `detail-result` 会触发真实 `detail-pages/generations`
- 会轮询 job
- 会拉真实 `detail-pages/results`
- 下载接真实 `detail-pages/download`

但仍有两个已知缺口：

- `detail-confirm` 目前是“策略示意预览”，不是后端真渲染的带水印详情图预览
- “记住品牌风格”只有前端本地占位，没有后端品牌风格库

## 5. 登录相关的正式口径

## 5.1 最终目标

甲方要的不是本地假草稿，也不是隐藏临时账号，而是：

- 前面的上传、识别、参数、策略、主图、详情图都是真实后端流程
- 用户可以先不登录
- 只在 `hd-result` 和 `detail-result` 两页允许用户可选登录/注册
- 登录成功后原地返回
- 把当前真实 session 认领到该账户
- 认领后资产进入该账户 `History`

## 5.2 当前现实边界

这一目标在“不改后端”的前提下不能真正成立，原因是：

- 当前 session、job、asset 都按正式用户归属
- `sessions` 相关接口普遍依赖 `get_current_user_id`
- 当前没有 guest session
- 当前没有 claim 接口

因此：

- 前端不能单独完成“游客真实跑完整链路，再在结果页登录并归档”的正式能力
- 正解必须是后端补 `guest session + claim`

- 登录入口只在 `hd-result` / `detail-result`

## 7. 延期项与后续联调重点

### 7.1 延期项

本轮不在前端强行落地：

- guest 真实上传
- guest 真实识别
- guest 真实参数 / 文案 / 策略 / 主图 / 详情图
- `hd-result` / `detail-result` 登录后真实 claim

### 7.2 后续联调重点

1. guest session + claim 方案是否被后端接受
2. 主图高清无水印是否补正式订单/交付接口
3. 详情图确认页是否补真预览接口
4. 白底图校验失败是否调整容错策略
5. 登录成功后 History 归档链路是否打通
