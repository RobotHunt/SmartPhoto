# 主图高清无水印_前端假逻辑与后端设计说明

## 1. 这份文档的用途

这份文档专门说明主图高清无水印这条链路里：

1. 当前前端已经做到什么
2. 当前哪些地方还是模板流程壳
3. 后端还缺哪些正式能力
4. 登录在这条链路里的正确位置是什么

---

## 2. 正确流程定义

主图主链路固定为：

`upload -> analyze -> platform -> generate -> strategy -> result -> payment -> hd-result`

关键说明：

- `result` 是主图结果页，不是登录入口页
- `payment` 是主图高清支付页
- `hd-result` 才是主图高清无水印结果页

登录相关的最终产品定义是：

- 登录不强制
- `payment` 不是登录入口
- 登录入口放在 `hd-result`
- `hd-result` 底部保留模板那行：
  - `登录账号，自动保存你的设计资产`
  - 右侧按钮 `登录 / 注册`
- 用户如果选择登录：
  - 登录成功后回到当前 `hd-result`
  - 不从头开始
  - 把当前真实 session 认领到账户
  - 认领后进入该账户 `History`
- 用户如果不登录，也不影响整条真实链路跑完

## 3. 当前前端已经落地的内容

## 3.1 `result` 页

当前前端已经保留真实主图能力：

- 单图重生
- 编辑文字后重生
- 整体优化
- 版本切换
- 选择哪些图片继续进入高清流程

并且会记录：

- `current_session_id`
- `current_result_version`
- `selected_asset_ids`

## 3.2 `payment` 页

当前前端已经能：

- 读取当前版本
- 读取当前选中的主图
- 展示支付模板页 UI
- 展示钱包 / 价格类信息

但这一步目前还不是正式的主图高清订单链路。

## 3.3 `hd-result` 页

当前前端已经能：

- 按当前版本和当前选图展示高清无水印结果页
- 大图尽量放大显示
- 用 `object-contain` 保证整图完整可见
- 支持单张下载
- 支持当前版本整包下载
- 底部保留登录提示区和 `登录 / 注册` 按钮

但这一步的“高清无水印结果”，本质上仍然是基于主图当前版本的交付展示壳，不是后端单独生成的一套新高清交付资产。

## 4. 当前后端真实缺口

## 4.1 缺少主图高清无水印专用订单能力

当前主图正式后端接口只有：

- `POST /api/v2/sessions/{session_id}/generations`
- `GET /api/v2/sessions/{session_id}/results`
- `GET /api/v2/sessions/{session_id}/download`

- `C:\Users\asus\Desktop\smartphoto_backend-main\smartphoto_backend-main\app\api\v2\sessions.py:2123`

当前没有：

- 主图高清订单创建接口
- 主图高清订单状态接口
- 主图高清交付结果接口
- 主图高清交付下载接口

## 4.2 缺少主图高清专用定价动作

当前定价规则只有：

- `generate_gallery`
- `generate_detail_page`
- `global_edit`
- `regenerate_asset`
- `regenerate_detail_panel`
- `regenerate_gallery`

涉及文件：

- `C:\Users\asus\Desktop\smartphoto_backend-main\smartphoto_backend-main\app\services\pricing.py`
- `C:\Users\asus\Desktop\smartphoto_backend-main\smartphoto_backend-main\app\core\config.py`

没有：

- `unlock_main_gallery_hd`

这会导致主图高清支付很难做成真正可记账、可追踪、可交付的正式能力。

## 4.3 缺少 guest session + claim，所以 `hd-result` 的登录提示还不能兑现完整业务含义

甲方要的是：

- 用户前面可以不登录，但跑真实后端流程
- 到 `hd-result` 再可选登录
- 登录成功后原地返回并把当前 session 归档到账户

但当前后端没有：

- guest session
- guest token
- claim 接口

所以：

- 现在 `hd-result` 的登录提示可以按模板摆出来
- 但真正的“登录后自动保存你的设计资产”仍然依赖后端补齐 guest session + claim

## 5. 推荐的后端正式方案

## 5.1 主图高清订单接口

建议新增：

### 创建订单

`POST /api/v2/sessions/{session_id}/main-gallery/hd-orders`

请求体建议：

```json
{
  "source_version_no": 3,
  "asset_ids": ["asset_a", "asset_b", "asset_c"]
}
```

### 查询订单

`GET /api/v2/sessions/{session_id}/main-gallery/hd-orders/{order_id}`

### 查询高清交付结果

`GET /api/v2/sessions/{session_id}/main-gallery/hd-results?order_id=hd_order_xxx`

### 下载高清交付结果

`GET /api/v2/sessions/{session_id}/main-gallery/hd-download?order_id=hd_order_xxx`

## 5.2 主图高清订单表

建议新增：

- `main_gallery_hd_orders`

建议字段：

- `id`
- `session_id`
- `user_id`
- `source_version_no`
- `selected_asset_ids_json`
- `status`
- `pricing_rule_id`
- `charged_credits`
- `wallet_transaction_id`
- `created_at`
- `paid_at`

## 5.3 定价动作

建议新增：

- `unlock_main_gallery_hd`

这样主图高清支付才能成为一个独立、可计费、可追踪、可交付的正式后端能力。

## 5.4 登录归档能力

如果最终要完全符合甲方要求，还必须叠加：

- guest session
- guest token
- claim 接口

也就是说：

- 主图高清订单能力解决的是“支付 / 交付”
- guest session + claim 解决的是“登录后归档到账户”

这两套能力缺一不可。

## 6. 后端补齐后，前端怎么收口

## `result`

保持不变：

- 继续维护 `selected_asset_ids`
- 继续维护 `current_result_version`

## `payment`

从当前的“页面内假支付”改成：

1. 提交 `source_version_no + asset_ids`
2. 创建主图高清订单
3. 成功后带 `order_id` 去 `hd-result`

## `hd-result`

从当前的“复用主图结果页展示”改成：

1. 读取 `order_id`
2. 拉真实 `hd-results`
3. 单张下载 / 整包下载走真实 `hd-download`
4. 底部继续保留模板登录提示
5. 登录成功后执行 claim，并原地返回

---

## 7. 当前前端实施口径

在后端未补齐前，本轮前端口径应统一为：

1. 主图高清无水印流程先按模板 UI 与交互落地
2. `result -> payment -> hd-result` 链路可以完整展示
3. 登录入口只放在 `hd-result`
4. `hd-result` 的登录提示先按模板保留
5. 真正的主图高清订单能力、登录后资产归档能力，等待后端补齐
