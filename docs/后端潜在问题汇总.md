## 问题 1：主图白底校验失败会直接拖垮整组主图生成

### 现象

在主图结果页 `/create/result`，会直接出现整组生成失败，常见报错为：

```text
white background validation failed after retry
```

前端有时还能看到类似这类诊断数据：

```text
{
  "edge_white_ratio": 0.9526,
  "outer_band_white_ratio": 0.9455,
  "major_components": 1,
  "passed": false,
  "retry_applied": true
}
```

### 影响

- 主图整组任务直接失败
- 用户看不到其余已成功生成的图片
- 结果页只能整体进入失败态

### 问题原因

#### 1. `white_bg` 槽位强制开启白底校验

涉及文件：

- `C:\Users\asus\Desktop\smartphoto_backend-main\smartphoto_backend-main\app\services\rule_packs.py:24`
- `C:\Users\asus\Desktop\smartphoto_backend-main\smartphoto_backend-main\app\services\rule_packs.py:25`

#### 2. 白底校验阈值较严

涉及文件：

- `C:\Users\asus\Desktop\smartphoto_backend-main\smartphoto_backend-main\app\services\white_bg.py:25`
- `C:\Users\asus\Desktop\smartphoto_backend-main\smartphoto_backend-main\app\services\white_bg.py:38`

当前规则：

- `edge_white_ratio >= 0.97`
- `outer_band_white_ratio >= 0.94`
- `major_components <= 1`

#### 3. 后端只自动重试一次，仍失败就直接抛错

涉及文件：

- `C:\Users\asus\Desktop\smartphoto_backend-main\smartphoto_backend-main\app\services\pipeline.py:1008`
- `C:\Users\asus\Desktop\smartphoto_backend-main\smartphoto_backend-main\app\services\pipeline.py:1036`
- `C:\Users\asus\Desktop\smartphoto_backend-main\smartphoto_backend-main\app\services\pipeline.py:1113`
- `C:\Users\asus\Desktop\smartphoto_backend-main\smartphoto_backend-main\app\services\pipeline.py:1141`

### 前端排查结论

这次问题不是前端重复触发生成导致的，排查结论如下：

- 当前前端没有 `StrictMode` 双调用
- 当前结果页没有发现重复提交 `/sessions/{id}/generations`
- 当前前端主图生成请求方式和旧前端一致

对应文件：

- `C:\Users\asus\Documents\New project\dev_free_work\client\src\main.tsx`
- `C:\Users\asus\Documents\New project\dev_free_work\client\src\pages\create\ResultStep.tsx`
- `C:\Users\asus\Documents\New project\dev_free_work\client\src\lib\api.ts`

### 可行解决方案

#### 方案 A：不要让单张白底图失败拖垮整组主图

建议优先级最高。

修改方向：

- 在 `pipeline.py` 中，不要因为 `white_bg` 一张失败就整组报错
- 把该槽位标成失败状态
- 允许其余已成功图片继续返回给前端

建议落点：

- `C:\Users\asus\Desktop\smartphoto_backend-main\smartphoto_backend-main\app\services\pipeline.py:1033-1038`
- `C:\Users\asus\Desktop\smartphoto_backend-main\smartphoto_backend-main\app\services\pipeline.py:1138-1143`

#### 方案 B：放宽白底阈值或做平台级配置

建议作为快速止血方案。

修改文件：

- `C:\Users\asus\Desktop\smartphoto_backend-main\smartphoto_backend-main\app\services\white_bg.py:38`

可选策略：

- 先把 `edge_white_ratio` 从 `0.97` 临时放宽到 `0.95`
- 或按平台区分阈值
- 或按品类区分阈值

#### 方案 C：错误契约改成前端可消费格式

不要再把原始诊断字典直接抛给前端。

当前位置：

- `C:\Users\asus\Desktop\smartphoto_backend-main\smartphoto_backend-main\app\services\pipeline.py:1034-1036`
- `C:\Users\asus\Desktop\smartphoto_backend-main\smartphoto_backend-main\app\services\pipeline.py:1139-1141`

## 问题 3：当前后端无法支持“游客真实跑完整链路，最后在 hd-result / detail-result 可选登录并归档”

### 现象

产品目标已经明确为：

#### 主图主链路

`upload -> analyze -> platform -> generate -> strategy -> result -> payment -> hd-result`

#### 详情图链路

`hd-result -> copywriting -> detail-confirm -> detail-result`

并且要求：

- 上述所有页面都走真实后端流程
- 用户前面可以不登录
- 登录入口只在：
  - `hd-result`
  - `detail-result`
- 用户如果在这两页登录成功：
  - 回到刚刚的页面
  - 把前面的真实 session 认领到账户
  - 然后进入该账户 `History`
- 用户如果不登录，也不影响整条真实生图链路跑完

### 影响

如果后端不支持这套能力：

- 前端无法在未登录状态下真实上传、识别、生成
- `hd-result` / `detail-result` 的“登录账号，自动保存你的设计资产”只能停留在 UI 文案层
- 登录后无法把前面的真实会话归档到当前账户

### 问题原因

#### 1. `SessionModel.user_id` 当前强绑定正式用户

涉及文件：

- `C:\Users\asus\Desktop\smartphoto_backend-main\smartphoto_backend-main\app\models\session.py:10`
- `C:\Users\asus\Desktop\smartphoto_backend-main\smartphoto_backend-main\app\models\session.py:13`

#### 2. `JobModel.user_id` 当前也强绑定正式用户

涉及文件：

- `C:\Users\asus\Desktop\smartphoto_backend-main\smartphoto_backend-main\app\models\job.py:10`
- `C:\Users\asus\Desktop\smartphoto_backend-main\smartphoto_backend-main\app\models\job.py:14`

#### 3. `sessions` 路由统一要求 `get_current_user_id`

涉及文件：

- `C:\Users\asus\Desktop\smartphoto_backend-main\smartphoto_backend-main\app\api\v2\sessions.py:321`
- `C:\Users\asus\Desktop\smartphoto_backend-main\smartphoto_backend-main\app\api\v2\sessions.py:347`

- `C:\Users\asus\Desktop\smartphoto_backend-main\smartphoto_backend-main\app\core\deps.py:11`

### 可行解决方案

#### 方案 A：新增 guest session + claim 机制

这是当前需求的正式解法。

##### session 模型建议

- `user_id` 改为可空
- 新增 `owner_type = guest | user`
- 新增 `guest_token_hash`
- 新增 `guest_token_expires_at`
- 新增 `claimed_at`

##### job 模型建议

- `user_id` 改为可空

- `POST /api/v2/guest/sessions/{id}/parameters/extract`

- `PUT /api/v2/guest/sessions/{id}/copy`

- 只有未登录用户才走 guest session

否则：

- 已登录用户中途退出时
- 其中间数据无法正常进入 `History`

## 问题 5：详情图生成会真实扣额度，额度不足时会直接 402 打断流程

### 现象

在 `/create/detail-result`，调用：

- `POST /api/v2/sessions/{session_id}/detail-pages/generations`

如果账户额度不足，会直接得到：

```text
402 Payment Required
insufficient credits
```

### 影响

- 详情图链路直接中断
- 很难向甲方完整演示 `detail-confirm -> detail-result`
- 当前失败态看起来像“功能坏了”，但根因其实是账户额度限制

### 问题原因

详情图生成是正式扣费动作，不是预览动作。

涉及文件：

- `C:\Users\asus\Desktop\smartphoto_backend-main\smartphoto_backend-main\app\api\v2\sessions.py:1753`
- `C:\Users\asus\Desktop\smartphoto_backend-main\smartphoto_backend-main\app\services\user_accounts.py:351`
- `C:\Users\asus\Desktop\smartphoto_backend-main\smartphoto_backend-main\app\services\user_accounts.py:366`
- `C:\Users\asus\Desktop\smartphoto_backend-main\smartphoto_backend-main\app\services\pricing.py:16`
- `C:\Users\asus\Desktop\smartphoto_backend-main\smartphoto_backend-main\app\core\config.py:74`

默认动作：

- `generate_detail_page = 16 credits`

### 可行解决方案

#### 方案 A：进入 detail-result 前先做额度校验

前端进入 `detail-result` 前先查：

- `/account/wallet`
- `/pricing-rules`

如果余额不足，提前在 `detail-confirm` 阶段提示，而不是进入失败页。

#### 方案 B：测试环境增加预览模式或免扣白名单

可选做法：

- `detail-pages/generations?mode=preview`
- 或 staging 测试用户免扣 `generate_detail_page`

#### 方案 C：前端演示兜底

仅用于甲方演示时，可在前端对 `402 insufficient credits` 增加详情图模拟结果页兜底，但不建议作为正式方案。

## 当前前端建议口径

在后端问题尚未修复前，前端建议统一使用以下口径：

1. 主图白底失败属于后端质检与容错问题
2. 主图高清无水印链路目前前端先按模板流程组织，正式订单能力待后端补齐
3. “游客先体验后登录”的正式能力必须依赖 guest session + claim
4. 详情图 402 多数是额度问题，不是前端接口参数错误
