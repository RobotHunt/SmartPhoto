## 5. 当前登录/注册逻辑与甲方目标需求的关系

### 5.1 甲方真正目标

甲方要求的登录逻辑是：

1. 上传、识别、参数、策略、主图、详情图，全都是真实后端流程
2. 用户在这些阶段可以不登录
3. 到 `hd-result` 和 `detail-result` 两个结果页时，页面底部出现“登录账号，自动保存你的设计资产”
4. 登录/注册不是强制，而是可选
5. 如果用户此时选择登录/注册：
   - 登录成功后回到刚刚的结果页
   - 不是从头来过
   - 前面这条真实 session 被认领到当前账号
   - 前面的主图/详情图进入该账号 `History`
6. 如果用户不登录，也不影响整条真实生图链路跑完

### 5.2 当前前端已经做到的部分

当前前端已做到：

- `hd-result` 和 `detail-result` 底部都已放模板风格的登录提示条
- 都有 `登录 / 注册` 按钮
- 点击后会跳登录页
- 登录页支持通过 `redirect` 参数跳回当前页面
- 登录成功后可以回到刚刚的 `hd-result` 或 `detail-result`
- 不需要从头开始流程

### 5.3 当前前端还做不到的部分

当前前端还做不到：

- 登录成功后，把前面那条 guest / 匿名真实 session 自动归档给当前账户
- 自动把主图、高清图、详情图资产写进该账户 `History`
- 已登录用户和未登录用户共用同一条 session 并在登录后 claim

原因：

- 这部分不是前端能单独完成的，需要后端支持 session 认领

## 6. 当前后端与甲方登录需求的差距

### 6.1 当前后端现状

当前后端整体上是“正式用户 session”模式：

- 多数核心接口都要求当前用户身份
- `session` 归属以正式 `user_id` 为主
- 目前没有完整 guest session 认领机制

这意味着：

- 如果真正要求用户“前面全程不登录，但流程依旧真实跑后端”
- 那么后端必须能先接受 guest 身份，再在登录后 claim 到 user

### 6.2 后端缺少的关键能力

后端当前缺少的核心能力可以概括为：

1. `guest session`
- 允许没有正式 `user_id` 的真实会话存在

2. guest 鉴权
- 需要 guest token 或等价机制访问该 session

3. `claim session`
- 用户登录后，把 guest session 认领到当前用户

4. 认领后的资产归档
- claim 后，当前 session 及相关主图、详情图资产进入正式用户 `History`

5. 已登录用户直连 user session
- 已登录用户不应走 guest session
- 应直接走正式用户链路

---

## 7. 推荐的后端实现方案

### 7.1 推荐方向

推荐后端做：

- `guest session + claim`

不推荐做：

- 前端本地假草稿充当真实 AI 链路
- 前端静默隐藏账号冒充游客
- 仅靠前端 sessionStorage 伪造游客登录体验

### 7.2 推荐模型能力

后端建议支持：

1. `sessions.user_id` 可空
2. `owner_type = guest | user`
3. `guest_token`
4. `claim` 接口
5. claim 后资产进入正式账户

### 7.3 推荐时机

真正的登录入口只保留在：

- `hd-result`
- `detail-result`

即：

- 不在首页强制登录
- 不在上传前强制登录
- 不在生成主图前强制登录
- 只在结果页提供可选登录/注册并认领资产

## 10. 一句话总结

当前前端已经把主图和详情图的大部分真实链路接到后端了，也已经把甲方模板要求的登录提示条先放到了 `hd-result` 和 `detail-result` 上，并支持登录后回到当前页。

但要完整满足甲方“先生图、后登录、登录后自动归档到账户 History”的要求，后端必须补 `guest session + claim` 这套能力；否则前端只能先把 UI 和回跳做出来，无法真正完成资产认领与归档。
