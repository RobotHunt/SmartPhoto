## 2. 当前前端已实现的真实链路
### 2.1 主图链路
当前主图链路按下面顺序组织：

`upload -> analyze -> platform -> generate -> strategy -> result -> payment -> hd-result`

对应状态：
- `upload`：真实创建 session、上传图片、删除图片、回填已上传图片。
- `analyze`：真实触发分析任务、轮询 job、读取 `analysis_snapshot`。
- `platform`：真实保存平台选择。
- `generate`：真实读取/保存参数，支持参数附件上传、参数提取、参考图上传。
- `strategy`：真实读取主图策略预览。
- `result`：真实主图生成、结果加载、版本切换、单图重生、整组整体优化。
- `payment`：当前按联调逻辑处理，不是真实支付闭环。
- `hd-result`：真实读取当前版本主图结果；下载能力受 `can_download` 控制。

### 2.2 详情图链路
当前详情图链路按下面顺序组织：

`hd-result -> copywriting -> detail-confirm -> detail-result`

对应状态：
- `copywriting`：作为详情图文案页，读取并回写当前 session 文案。
- `detail-confirm`：真实调用详情图策略预览接口。
- `detail-result`：真实触发详情图生成、轮询 job、读取详情图结果、版本切换、下载能力受 `can_download` 控制。

## 4. 当前 guest-first 已落地的前端表现
### 4.1 不再要求首页先登录
- 首页进入创建流程时，不再强制跳登录页。
- 可以直接进入 `upload`。

### 4.2 hd-result 与 detail-result 已补登录/注册入口
当前两页都已经先按甲方模板补了登录提示条：
- 文案：登录账号，自动保存你的设计资产
- 按钮：登录 / 注册

当前前端已实现：
- 点击后跳 `login`
- 自动带 `redirect` 回跳参数
- 登录成功后回到刚刚的 `hd-result` 或 `detail-result`

### 4.3 登录后认领 session
当前前端已实现：
- 登录成功回到 `hd-result` / `detail-result` 后，如果 URL 带 `?claim=1`
- 会主动调用：
  - `POST /api/v2/guest/sessions/{session_id}/claim`

说明：
- 这一步现在已经按新后端接口接上。
- 认领成功后前端会切换到账号态，并刷新 `can_download`。

## 5. 当前仍与甲方最终要求存在差距的地方
### 5.1 登录入口位置
甲方最终要求：
- 登录入口只放在 `hd-result` 和 `detail-result`
- 前面流程全部允许未登录 guest 真实跑完

当前状态：
- 结果页后的 guest 逻辑已开始对齐
- 但前面部分页面与导航里仍残留历史登录导向，需要持续清理

### 5.2 History 归档
甲方要求：
- 如果用户在 `hd-result` / `detail-result` 登录成功
- 前面的真实 session 应归档到当前账户 `History`

当前状态：
- 前端已经接了 `claim`
- 但是否完整进入 `History`，取决于后端认领逻辑和 `/account/assets` 是否按认领后 session 返回

### 5.3 detail-confirm 真预览图
甲方视觉上更像“详情图已经有预览”。

当前状态：
- `detail-confirm` 里展示的是：
  - 真实 `detail strategy preview` 数据
  - 叠加主图结果 / 上传图的占位示意
- 不是后端真渲染出来的详情图缩略预览

## 6. 后端还需要重点保障的能力
### 6.1 guest 真全流程可跑
已从新后端文档中确认，前端当前依赖后端保持如下能力可用：
- guest 可创建 session
- guest 可上传图片
- guest 可分析、提取参数、生成文案、生成主图、生成详情图、做全局修改和单图重生

### 6.2 claim
后端需保证以下能力稳定可用：
- `POST /api/v2/guest/sessions/{session_id}/claim`
- 要求：
  - 用户已登录
  - 浏览器仍保留该 guest 的 cookie
  - claim 后仍复用原 `session_id`
  - claim 幂等

### 6.3 会话快照字段
前端当前已经消费这些字段：
- `auth_mode`
- `can_download`
- `can_continue_editing`
- `login_required_actions`
- `latest_generate_job_id`
- `latest_detail_generate_job_id`

后端需要保证这些字段持续稳定返回。

## 7. 当前不作为问题处理的点
以下点当前已与产品侧达成阶段性共识，暂不处理：
- `payment -> hd-result` 先按扣额度/联调逻辑，不追求真实支付闭环。
- `detail-confirm` 的卡片预览先允许使用占位示意，不强求真 AI 预览图。
- 登录逻辑最终产品位置已确定，但如有更细回跳边界，可等下一轮再细化。
