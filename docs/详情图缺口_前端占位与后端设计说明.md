# 详情图缺口_前端占位与后端设计说明

## 1. 这份文档的用途

这份文档只聚焦详情图链路，说明：

1. 当前哪些能力已经接上真实后端
2. 哪些能力仍然是前端占位
3. 登录在详情图链路里的正确位置是什么
4. 后端还缺哪些正式能力

---

## 2. 正确流程定义

详情图链路固定为：

`hd-result -> copywriting -> detail-confirm -> detail-result`

关键说明：

- `copywriting` 在这里是详情图文案确认页
- `detail-confirm` 是详情图方案确认页
- `detail-result` 是详情图结果页

登录相关的最终定义是：

- 登录不强制
- 登录入口不放在 `copywriting`、`detail-confirm`
- 登录入口放在 `detail-result`
- `detail-result` 底部保留模板那行：
  - `登录账号，自动保存你的设计资产`
  - 右侧按钮 `登录 / 注册`
- 用户如果选择登录：
  - 登录成功后回到 `detail-result`
  - 把当前真实 session 认领到账户
  - 主图与详情图一起进入该账户 `History`
- 用户如果不登录，也不影响详情图链路真实跑完

---

## 3. 当前已经接上的真实后端能力

当前后端已经存在真实接口：

- `POST /api/v2/sessions/{session_id}/detail-pages/strategy/preview`
- `POST /api/v2/sessions/{session_id}/detail-pages/generations`
- `GET /api/v2/sessions/{session_id}/detail-pages/results`
- `GET /api/v2/sessions/{session_id}/detail-pages/download`

这意味着当前前端可以真实接：

- `detail-confirm` 的策略预览
- `detail-result` 的详情图生成
- 详情图结果查询
- 详情图下载

因此，详情图链路本身并不是纯模板 mock，而是已经有真实后端能力基础。

## 4. 当前仍是前端占位的部分

## 4.1 `detail-confirm` 缺少“真预渲染带水印详情图预览”

### 当前现状

模板里的 `detail-confirm` 更像是：

- 已经有 6-8 张带水印详情图预览
- 用户确认后再正式生成高清详情图

但当前后端 `detail-pages/strategy/preview` 返回的是：

- `detail_strategy_preview.panel_plan`
- `panel_label`
- `panel_type`
- `copy_lines`
- `copy_blocks`
- `layout_notes`

它返回的是“策略规划”，不是“真渲染好的详情图预览图 URL”。

### 当前前端做法

为了不阻塞流程，前端现在是：

- 先取真实 `detail-pages/strategy/preview`
- 再拿主图结果或原图做参考底图
- 把 `panel_label / copy_lines / layout_notes` 叠到卡片里
- 生成一个“详情图方案示意页”

所以当前 `detail-confirm` 是：

- **真实策略预览**
- 但**不是真实带水印详情图预览**

### 后端建议方案

建议新增一条详情图预览渲染接口，例如：

- `POST /api/v2/sessions/{session_id}/detail-pages/previews`

返回建议：

```json
{
  "session_id": "xxx",
  "preview_version": 1,
  "panels": [
    {
      "panel_id": "product_showcase",
      "panel_label": "产品展示",
      "preview_image_url": "https://...",
      "watermark_applied": true
    }
  ]
}
```

- `C:\Users\asus\Desktop\smartphoto_backend-main\smartphoto_backend-main\app\services\detail_pages.py`

## 4.2 `记住品牌风格` 目前只有前端占位，没有品牌风格库

### 当前现状

模板 `detail-result` 底部有“记住品牌风格”。

当前后端没有：

- 品牌风格库
- session 与品牌风格绑定能力
- 下次生成时自动继承品牌风格的能力

### 当前前端做法

当前前端已经改成：

- 保留模板入口
- 可以填一些本地数据
- 给出保存成功反馈
- 但不回传后端

### 后端建议方案

建议新增品牌风格实体，例如：

- `brand_style_profile`

建议字段：

- `id`
- `user_id`
- `brand_name`
- `style_summary`
- `copy_preferences`
- `prompt_preferences`
- `reference_image_manifest`
- `created_at`
- `updated_at`

建议接口：

- `POST /api/v2/brand-styles`
- `GET /api/v2/brand-styles`
- `GET /api/v2/brand-styles/{id}`

建议修改位置：

- `app\models\`
- `app\api\v2\`
- `app\services\detail_pages.py`
- `app\services\upstream.py`

## 4.3 `detail-result` 的单 panel 重生目前没有正式接口

### 当前现状

模板交互容易让人理解为支持：

- 某一张详情图单独重生

但当前后端确认存在的是：

- 整组详情图生成
- 整组结果查询
- 整组下载

### 当前前端做法

- `detail-confirm` 的“重新生成”目前更接近重新请求整个策略预览
- `detail-result` 的“重新生成”目前更接近重新触发整组详情图生成

请求体建议：

## 4.4 `detail-result` 的登录提示还不能兑现真正的“自动保存到账号”

### 正确目标

`detail-result` 底部登录提示的真实含义应该是：

- 用户此时可以可选登录 / 注册
- 登录成功后回到当前 `detail-result`
- 把前面的真实 session 认领到账户
- 主图和详情图一起进入该账户 `History`

### 当前缺口

当前后端没有：

- guest session
- guest token
- claim 接口

所以：

- 这条登录提示可以按模板摆出来
- 但真正的“自动保存你的设计资产”仍依赖后端 guest session + claim

---

## 5. 当前建议

在后端未补齐以上能力前，前端建议保持：

### `detail-confirm`

- 使用真实 `detail_strategy_preview`
- 用主图 / 原图做“示意预览”
- 不对外宣称这里已经是真预渲染带水印详情图

### `detail-result`

- 使用真实 `detail-pages/generations`
- 使用真实 `detail-pages/results`
- 使用真实 `detail-pages/download`

### `记住品牌风格`

- 先按模板保留
- 允许填写本地数据
- 但明确暂不回传后端

### 登录提示

- 只放在 `detail-result`
- 按模板展示
- 但对内明确：真正归档到账户仍待后端补 guest session + claim

---

## 6. 一句话结论

详情图链路当前已经不是纯假页：

- 策略预览、生成、结果、下载都能接真实后端

但想完全达到甲方模板语义，还缺三类后端能力：

1. 真预渲染详情图预览
2. 品牌风格库
3. guest session + claim
