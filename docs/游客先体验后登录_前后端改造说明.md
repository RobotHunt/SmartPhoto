# 游客先体验后登录：前后端改造说明

## 1. 目标定义
本项目要实现的不是“本地假草稿”，也不是“隐藏临时账号”，而是下面这套真实链路：

1. 用户在未登录状态下，也可以真实跑完整条创作流程。
2. 上传、识别、参数、策略、主图、详情图都是真实后端接口与真实 AI 结果。
3. 用户不被强制要求提前登录。
4. 登录 / 注册入口只放在两个页面：
   - `hd-result`
   - `detail-result`
5. 如果用户在这两个页面选择登录 / 注册：
   - 登录成功后回到刚刚所在的页面
   - 不从头来过
   - 把前面的真实 session 认领到当前账户
   - 认领后主图、高清图、详情图都进入该账户 `History`
6. 如果用户不登录，也不影响整条真实生图链路跑完。

一句话总结：

**真实 guest session 全流程创作，可选在 `hd-result / detail-result` 登录并认领到账号。**

## 2. 正确的页面流程
### 2.1 主图链路
主图正式链路为：

`upload -> analyze -> platform -> generate -> strategy -> result -> payment -> hd-result`

说明：
- `generate` 后直接进入 `strategy`
- `result` 是主图结果页
- `payment` 是高清支付页壳
- `hd-result` 是高清无水印结果页

### 2.2 详情图链路
详情图链路为：

`hd-result -> copywriting -> detail-confirm -> detail-result`

说明：
- `copywriting` 在这里是详情图文案确认页
- `detail-confirm` 是详情图方案确认页
- `detail-result` 是详情图结果页

## 3. 登录入口的正确位置
登录 / 注册入口只放在两个结果页：

1. `hd-result`
2. `detail-result`

这两个页面底部都应按甲方模板放一条提示：

- 文案：登录账号，自动保存你的设计资产
- 按钮：登录 / 注册

它们都是**可选入口**，不是强制门禁。

## 4. 登录后的正确行为
如果用户在 `hd-result` 或 `detail-result` 点击登录 / 注册：

1. 进入登录页或注册页
2. 登录成功后回到刚刚的页面
3. 不允许重新从 `upload` 开始
4. 登录成功后立即认领当前 guest session
5. 认领成功后，当前 session 及其主图/详情图资产进入该用户 `History`

## 5. 当前不能接受的错误方案
### 5.1 本地假草稿
错误原因：
- 登录前没有真实 AI 结果
- 登录前没有真实 session
- 不符合“前面的上传、识别、生成都是真实后端链路”的要求

### 5.2 隐藏临时账号
错误原因：
- 前面 session 实际归属隐藏账号
- 后面用户再登录时无法可靠迁移
- History 与资产归属会分裂

### 5.3 过早登录
以下都不是正确位置：
- 首页先登录
- 上传前先登录
- 生图前先登录
- `result` 页要求登录
- `payment` 页要求登录

正确位置只应是：
- `hd-result`
- `detail-result`

## 6. 为什么只改前端做不到
如果没有后端配合，只改前端做不到以下目标：

- 未登录先跑真实上传
- 未登录先拿到真实识别结果
- 未登录先生成真实主图与详情图
- 到结果页再登录
- 登录后把前面这条真实 session 自动归到当前账户

原因是：
- session 当前必须有合法访问身份
- 历史资产需要真正归属到账户
- 没有后端 `claim` 或等价迁移能力时，前端无法自行完成归档

## 7. 后端的正式推荐方案
最佳方案是：

**guest session + claim**

### 7.1 Session 模型
建议具备以下能力：
- `user_id` 可空
- guest 可以创建真实 session
- user 可以拥有正式 session
- claim 后原 session 仍保留原 `session_id`

### 7.2 Guest 访问方式
后端应支持：
- 浏览器 guest cookie
- guest 可以在当前 session 内持续创作
- guest 不可下载、不可进入 `/account/assets`

### 7.3 Claim 接口
关键接口：

`POST /api/v2/guest/sessions/{session_id}/claim`

要求：
- 用户已登录
- 浏览器里仍保留对应 guest cookie
- claim 幂等
- claim 后当前 session 归属到当前用户

### 7.4 Session 快照字段
前端当前依赖以下字段来决定页面表现：
- `auth_mode`
- `can_download`
- `can_continue_editing`
- `login_required_actions`
- `latest_generate_job_id`
- `latest_detail_generate_job_id`
- `latest_result_version`
- `detail_latest_result_version`

## 8. 当前前端已做的准备
前端已经先做好的部分：

1. `hd-result` 页底部已加模板式登录/注册提示条
2. `detail-result` 页底部已加模板式登录/注册提示条
3. 登录页支持 `redirect` 回跳
4. 登录成功后能返回刚刚的 `hd-result` / `detail-result`
5. 已接入：
   - `session snapshot`
   - `can_download`
   - `claimGuestSession`

说明：
- 也就是说，前端现在已经具备承接后端 `guest-first + claim` 的基础结构。
- 一旦后端这套能力稳定，前端不需要再改产品逻辑方向，只需继续完善细节。

## 9. 当前需要后端重点确认的事项
请后端重点确认下面这些点：

1. guest 是否能真实跑完整条主图与详情图链路
2. `claim` 是否只认领当前 session，而不是整浏览器 guest 身份
3. `claim` 是否复用原 `session_id`
4. `claim` 后 `/account/assets` 是否能看到该 session
5. `can_download` 与 `login_required_actions` 是否始终准确
6. 已登录用户是否始终直接走 user 链路，而不会误降级为 guest

## 10. 阶段性共识
当前已经明确的阶段性结论：

- 登录不是进入系统的门票
- 登录不是前置门禁
- 登录是结果页上的“资产归档入口”
- guest 真实创作链路优先于账户体系
- 只要后端 guest + claim 能力稳定，前端即可按甲方要求完成整体体验
