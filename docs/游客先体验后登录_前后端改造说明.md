# 游客先体验后登录_前后端改造说明

## 1. 最终目标

甲方要的不是本地假草稿，也不是隐藏临时账号，而是下面这套真实链路：

1. 用户在未登录状态下，也可以跑完整的真实创作流程
2. 上传、识别、参数、策略、主图、详情图，全部都是真实后端接口与真实 AI 结果
3. 不强制用户登录
4. 登录/注册入口只放在两个页面：
   - `hd-result`
   - `detail-result`
5. 用户如果在这两个页面选择登录/注册：
   - 登录成功后回到刚刚所在页面
   - 不从头来过
   - 把前面的真实 session 认领到当前账户
   - 认领后主图、高清图、详情图等资产进入该账户 `History`
6. 用户如果不登录，也不影响整条真实生图链路跑完

一句话概括：

**真实 guest session 全流程创作，可选在 `hd-result` / `detail-result` 登录并认领资产。**

---

## 2. 以本说明为准的正确流程

## 2.1 主图主链路

主图正式链路固定为：

`upload -> analyze -> platform -> generate -> strategy -> result -> payment -> hd-result`

说明：

- `generate` 后面直接到 `strategy`
- 不是 `generate -> copywriting`
- `result` 是主图结果页
- `payment` 是高清支付页
- `hd-result` 是高清无水印结果页

## 2.2 详情图链路

详情图链路固定为：

`hd-result -> copywriting -> detail-confirm -> detail-result`

说明：

- `copywriting` 在这里是详情图文案确认页
- `detail-confirm` 是详情图方案确认页
- `detail-result` 是详情图结果页

## 2.3 登录入口位置

登录 / 注册入口只放在两个页面：

1. `hd-result`
2. `detail-result`

这两个页面底部都应按甲方模板保留一行提示：

- `登录账号，自动保存你的设计资产`
- 右侧有 `登录 / 注册` 按钮

登录是可选动作，不是强制门禁。

## 2.4 登录后的页面回跳与认领

如果用户在 `hd-result` 或 `detail-result` 点击登录 / 注册：

1. 进入登录 / 注册页
2. 登录成功后回到刚刚的页面
   - 从 `hd-result` 进去，就回 `hd-result`
   - 从 `detail-result` 进去，就回 `detail-result`
3. 不允许从头重新走流程
4. 需要把当前真实 session 认领到当前账户
5. 认领成功后，这条 session 及其资产进入该账户的 `History`

---

## 3. 不能接受的错误方案

以下方案都不是本项目的正式目标：

## 3.1 本地假草稿

即：

- 登录前只在前端本地保存图片和表单
- 登录后再“回放”

问题：

- 登录前没有真实 AI 识别结果
- 登录前没有真实 session
- 不符合甲方“前面流程必须是真实后端流程”的要求

## 3.2 隐藏临时账号

即：

- 前端偷偷注册 / 登录一个临时账号
- UI 假装未登录
- 后面用户再登录自己的账号

问题：

- 前面的 session 实际归属临时账号
- 后面没有可靠方式把 session 迁移给真实账号
- `History` 和资产归属会分裂

## 3.3 过早登录

以下都不是正确位置：

- 首页先登录
- 上传前强制登录
- 生图前强制登录
- `result` 页面就要求登录
- `payment` 页面要求登录

本项目里，登录入口的正确位置只有：

- `hd-result`
- `detail-result`

---

## 4. 为什么只改前端做不到

当前后端实现决定了：

1. `sessions` 相关接口普遍要求已登录用户身份
2. session 当前是按 `user_id` 强归属的
3. 没有 guest session
4. 没有 guest -> user 的 claim / 认领机制

因此，仅改前端无法实现下面这个正式目标：

- 未登录用户先跑真实上传
- 先拿到真实 AI 识别、真实主图、真实详情图
- 到 `hd-result` / `detail-result` 再登录
- 登录后把前面的真实 session 归档到当前账户

前端最多只能做 UI 和跳转模拟，做不到真实归档。

---

## 5. 后端必须支持的正式方案

最佳方案是：

**guest session + claim**

## 5.1 session 模型调整

建议 `sessions` 支持以下字段：

- `user_id`：允许为空
- `owner_type`：`guest | user`
- `guest_token_hash`
- `guest_token_expires_at`
- `claimed_at`

含义：

- 未登录用户创建的真实创作会话属于 `guest`
- 登录后通过 claim 变成 `user`

## 5.2 jobs 模型调整

建议 `jobs` 支持：

- `user_id` 可空
- guest 阶段的分析 / 参数 / 文案 / 主图 / 详情图任务可以先不绑定正式用户

## 5.3 guest 鉴权方式

建议增加一种 guest 访问凭证：

- `X-Guest-Token`

或：

- guest JWT

要求：

- 只能访问自己的 guest session
- 不能跨 session 访问
- 具备有效期

## 5.4 guest 可走的真实接口范围

guest session 需要能走完整真实链路，至少包括：

- 创建 session
- 上传图片
- 触发分析
- 获取分析结果
- 平台选择
- 参数附件上传
- 参数提取
- 获取 / 更新参数
- 获取 / 更新文案
- 文案重生
- 主图策略预览
- 主图生成
- 主图结果查询
- 高清结果页相关上下文
- 详情图文案
- 详情图策略预览
- 详情图生成
- 详情图结果查询

## 5.5 claim 接口

必须新增：

`POST /api/v2/guest/sessions/{session_id}/claim`

职责：

- 要求当前用户已登录
- 校验 guest token
- 将当前 guest session 认领到当前用户
- 写入 `user_id`
- 把 `owner_type` 改为 `user`
- 记录 `claimed_at`

## 5.6 History 归档规则

### 未 claim 前

- guest session 不进入正式 `/account/assets`
- 不出现在用户 `History`

### claim 后

- 当前 session 进入该用户 `History`
- 主图、高清结果、详情图结果都归属到该用户
- 后续下载、继续编辑、查看版本都用正式用户权限

---

## 6. 结果页阶段的正确登录逻辑

## 6.1 hd-result

页面底部固定存在模板式登录提示：

- 提示文案：登录账号，自动保存你的设计资产
- 按钮：登录 / 注册

如果点击：

1. 进入登录 / 注册
2. 成功后回 `hd-result`
3. 自动 claim 当前 session
4. claim 成功后本次创作进入当前用户 `History`

## 6.2 detail-result

逻辑与 `hd-result` 相同：

1. 页面底部保留模板式登录提示
2. 登录 / 注册按钮可选点击
3. 成功后回 `detail-result`
4. 自动 claim 当前 session
5. 详情图与主图一起进入当前用户 `History`

---

## 7. 已登录用户的规则

这条必须明确：

**已登录用户始终优先走正式 user session，不走 guest session。**

原因：

- 已登录用户如果中途退出，未完成记录才会正确进入 `History`
- 避免把已登录用户错误降级成 guest
- 避免 session 归属混乱

因此系统需要分两条入口：

### 未登录用户

- 创建 guest session
- 走真实 guest 创作链路
- 可在 `hd-result` / `detail-result` 时登录并 claim

### 已登录用户

- 直接创建正式 user session
- 全流程都归属到当前用户

---

## 8. 前端需要配合的能力

在后端 guest session + claim 完成后，前端要做这些事情：

## 8.1 路由与上下文

需要稳定维护：

- `session_id`
- `owner_type`
- `guest_token`
- 当前页面来源
- 当前查看版本
- 当前高清结果上下文

## 8.2 登录回跳

登录入口只出现在：

- `hd-result`
- `detail-result`

登录时要带回跳信息：

- 来自哪个页面
- 当前 `session_id`
- 当前版本

## 8.3 登录成功后的动作

登录成功后前端应自动执行：

1. 调 `claim`
2. claim 成功后重新拉当前 session snapshot
3. 回到原页面
4. 给出“已保存到账号”的正向提示

## 8.4 不强制用户登录

前端不能在下面这些页面强插登录门禁：

- `upload`
- `analyze`
- `platform`
- `generate`
- `strategy`
- `result`
- `payment`

## 9. 推荐后端接口清单

以下是建议清单，命名可微调，但职责应完整：

## 9.1 guest session 基础

- `POST /api/v2/guest/sessions`
- `GET /api/v2/guest/sessions/{id}`

## 9.2 guest 图片与分析

- `POST /api/v2/guest/sessions/{id}/images`
- `GET /api/v2/guest/sessions/{id}/images`
- `POST /api/v2/guest/sessions/{id}/analysis`
- `GET /api/v2/guest/sessions/{id}/analysis`

## 9.3 guest 平台与参数

- `PUT /api/v2/guest/sessions/{id}/platform-selection`
- `GET /api/v2/guest/sessions/{id}/parameters`
- `PUT /api/v2/guest/sessions/{id}/parameters`
- `POST /api/v2/guest/sessions/{id}/parameters/extract`

## 9.4 guest 文案与策略

- `GET /api/v2/guest/sessions/{id}/copy`
- `PUT /api/v2/guest/sessions/{id}/copy`
- `POST /api/v2/guest/sessions/{id}/copy/regenerate`
- `POST /api/v2/guest/sessions/{id}/strategy/preview`

## 9.5 guest 主图与详情图结果

- `POST /api/v2/guest/sessions/{id}/generations`
- `GET /api/v2/guest/sessions/{id}/results`
- `POST /api/v2/guest/sessions/{id}/detail-pages/strategy/preview`
- `POST /api/v2/guest/sessions/{id}/detail-pages/generations`
- `GET /api/v2/guest/sessions/{id}/detail-pages/results`

## 9.6 claim

- `POST /api/v2/guest/sessions/{id}/claim`

---

## 10. 给后端的执行摘要

这次登录改造的关键，不是“做一个游客态页面”，而是：

1. 未登录用户也能跑完整真实 AI 创作流程
2. 登录入口只出现在 `hd-result` 和 `detail-result`
3. 登录不是门禁，而是资产归档入口
4. 登录成功后要 claim 当前真实 session
5. claim 后该 session 必须进入用户 `History`

如果没有 `guest session + claim`，这套需求就无法正确落地。

---

## 11. 当前前端实施策略

在后端未完成前，本轮前端不强行落半套假方案。

原因：

- 本地草稿是假链路
- 隐藏账号是脏方案
- 强行把登录放到错误页面会继续偏离需求

当前应以本文档为准，等待后端确认后再正式推进。
