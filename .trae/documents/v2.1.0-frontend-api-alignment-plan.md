# v2.1.0 前端接口对齐实施计划

> 基于 `v2.1.0-frontend-api-changes.md` 文档，对齐前端代码与后端 v2.1.0 接口变更

---

## 变更概览

| # | 变更项 | 当前状态 | 目标状态 | 涉及文件 |
|---|--------|---------|---------|---------|
| 1 | 详情图结果新增 `copy_blocks` | `DetailPanel` 类型无 `copy_blocks` 字段 | 解析并使用 `copy_blocks` 回显真实文案 | `api.ts`, `DetailResultStep.tsx` |
| 2 | `editAssetText` 支持详情图 | 详情图用 `regenerate` 修文字 | 迁移为 `editAssetText` + `copy_blocks` | `DetailResultStep.tsx`, `api.ts` |
| 3 | `instruction` 支持主题方向 | 已有 `INSTRUCTION_COPY_MAP`，仅主图 | 确认主图流程完整，评估详情图是否需支持 | `ResultStep.tsx` |
| 4 | `regenerateCopy` 后必须 `PUT /copy` | `saveCopy` 已定义但从未调用 | 在文案重新生成后调用 `saveCopy` | `CopywritingStep.tsx`, `StrategyStep.tsx` |
| 5 | 详情图编辑框文案来源 | 编辑框用策略预览文案 | 用 `copy_blocks` 替代策略预览文案 | `DetailResultStep.tsx` |

## 详细步骤

### 任务 1：扩展 `DetailCopyBlocks` 类型 & 详情图结果新增 `copy_blocks`

**问题**：后端 v2.1.0 在 `GET detail-pages/results` 的 `panels[]` 中新增了 `copy_blocks` 字段，但前端 `DetailPanel` 类型中缺少该字段，`normalizeDetailResults` 也没有提取它。

**对比**：主图 `MainGalleryCopyBlocks` 有 `headline`, `supporting`, `proof_lines`, `matrix_lines`；但详情图新增的 `copy_blocks` 还包含 `bullet_points` 和 `cta_line`。

**修改**：

1. **`api.ts`** — 新增 `DetailCopyBlocks` 接口：
   ```typescript
   export interface DetailCopyBlocks {
     headline: string;
     supporting: string;
     bullet_points: string[];
     proof_lines: string[];
     matrix_lines: string[];
     cta_line: string;
   }
   ```

2. **`DetailResultStep.tsx`** — `DetailPanel` 类型新增 `copy_blocks` 字段：
   ```typescript
   copy_blocks?: DetailCopyBlocks | null;
   ```

3. **`DetailResultStep.tsx`** — `normalizeDetailResults` 中提取 `copy_blocks`：
   ```typescript
   copy_blocks: panel?.copy_blocks ?? null,
   ```

4. **`api.ts`** — `editAssetText` 方法的 `copyBlocks` 参数类型从 `Partial<MainGalleryCopyBlocks>` 扩展为同时接受详情图文案块（使用联合类型或 `Partial<DetailCopyBlocks>`），或者新增一个 `editDetailAssetText` 专用方法（推荐后者，更清晰）。

---

### 任务 2：详情图文字编辑迁移到 `editAssetText`

**问题**：当前 `DetailResultStep.tsx` 的 `saveText` 方法使用 `assetAPI.regenerate(id, instruction)` 修改详情图文字，这会导致整图重绘。v2.1.0 的 `editAssetText` 已支持 `detail_page`，可仅替换文字区域、保持原图构图。

**当前代码** (`DetailResultStep.tsx` L480-501)：
```typescript
const instruction = `将图片上的标注文字修改为：${img.text}`;
const response = await assetAPI.regenerate(id, instruction);
```

**修改**：

1. **`api.ts`** — 新增 `editDetailAssetText` 方法（或修改 `editAssetText` 签名支持详情图类型）：
   ```typescript
   editDetailAssetText(
     assetId: string,
     copyBlocks: Partial<DetailCopyBlocks>,
     instruction?: string | null,
   ) {
     return apiFetch(`/assets/${assetId}/edit-text`, {
       method: 'POST',
       body: JSON.stringify({
         copy_blocks: copyBlocks,
         instruction: instruction ?? null,
       }),
     });
   },
   ```

2. **`DetailResultStep.tsx`** — 修改 `DetailImage` 接口，将 `text?: string` 替换为结构化的 `copy_blocks`：
   ```typescript
   interface DetailImage {
     id: string;
     label: string;
     url: string;
     copy_blocks?: DetailCopyBlocks | null;  // 替代原来的 text?: string
     editOpen: boolean;
     isRegenerating: boolean;
     // ...其他字段不变
   }
   ```

3. **`DetailResultStep.tsx`** — 重写 `saveText` 方法，改用 `editAssetText`：
   ```typescript
   const saveText = async (id: string) => {
     const img = images.find((item) => item.id === id);
     if (!img?.copy_blocks) return;
     // ...
     const response = await assetAPI.editDetailAssetText(id, img.copy_blocks);
     // ...
   };
   ```

4. **`DetailResultStep.tsx`** — 重构编辑 UI：将单个 `text` 输入框替换为结构化编辑面板（headline / supporting / bullet_points / proof_lines / matrix_lines / cta_line），参考主图 `ResultStep.tsx` 和 `HDResultStep.tsx` 的编辑面板实现。

### 任务 3：`instruction` 主题方向注入（主图已有，确认完整性）

**问题**：v2.1.0 明确了 `instruction` 字段支持主题方向关键词的注入机制。前端 `ResultStep.tsx` 已有 `INSTRUCTION_COPY_MAP` 做关键词→文案映射。

**当前实现分析**：
- `ResultStep.tsx` L66-71 有 `INSTRUCTION_COPY_MAP`（宠物净化、除甲醛、除菌、静音）
- L492-511 在 `handleRegen` 中检测指令是否匹配关键词，匹配则更新 `copy_blocks`
- 后端 v2.1.0 说明当 instruction 不是结构化指令时，会自动注入到 prompt 的 goal/subject/constraints/copy_blocks

**结论**：前端的 `INSTRUCTION_COPY_MAP` 是**前端侧的预览文案替换**，后端侧也有独立的主题注入逻辑。两者并不冲突，但前端映射只是为了让用户在预览中看到变化，最终效果仍由后端控制。

**修改**：

1. **`ResultStep.tsx`** — 检查 `INSTRUCTION_COPY_MAP` 覆盖的主题是否足够，考虑是否需要扩展（目前覆盖了宠物净化/除甲醛/除菌/静音，但后端可能支持更多品类关键词）。**建议**：当前映射表作为"预览提示"已足够，无需大幅修改；后端会自行处理未匹配的主题关键词。

2. **详情图** — 评估是否需在 `DetailResultStep.tsx` 中也支持主题方向 instruction。当前详情图重新生成走 `assetAPI.regenerate`，instruction 可直接传入主题关键词，后端已支持。**建议**：在详情图"重新生成"操作中允许用户输入主题关键词 instruction（如果当前 UI 支持的话）。

### 任务 4：`regenerateCopy` 后必须调用 `PUT /copy` 保存

**问题**：v2.1.0 文档明确指出 `regenerateCopy` 是异步接口，返回 `generated_fields`，但**必须**调用 `PUT /copy` 将 `generated_fields` 写入 `confirmed_copy` 才会生效。当前前端代码**从未调用 `saveCopy`**。

**当前流程**（`CopywritingStep.tsx` L270-280 & `StrategyStep.tsx` L79-81）：
```
POST /copy/regenerate → job_id → 轮询 → GET /copy 读取结果
```
**缺少**：`PUT /copy` 写入 confirmed_copy。

**修改**：

1. **`api.ts`** — `regenerateCopy` 的返回类型需要包含 `generated_fields`：
   ```typescript
   regenerateCopy(sessionId: string, targets: string[], instruction?: string) {
     const body: any = { targets, based_on_current_values: true };
     if (instruction) body.instruction = instruction;
     return apiFetch<{ job_id: string }>(`/sessions/${sessionId}/copy/regenerate`, {
       method: 'POST', body: JSON.stringify(body),
     });
   },
   ```

2. **`api.ts`** — `saveCopy` 方法已有（L532-534），但参数类型是 `any`，建议明确：
   ```typescript
   saveCopy(sessionId: string, data: Record<string, any>) {
     return apiFetch(`/sessions/${sessionId}/copy`, { method: 'PUT', body: JSON.stringify(data) });
   },
   ```

3. **`CopywritingStep.tsx`** — 在 `regenerateCopy` 轮询完成后，调用 `saveCopy` 保存结果：
   ```typescript
   // 当前代码（L270-280）:
   const { job_id } = await sessionAPI.regenerateCopy(sessionId, ALL_TARGETS);
   await jobAPI.pollUntilDone(job_id, ...);
   const result = await sessionAPI.getCopy(sessionId);
   
   // 需要新增：
   if (result) {
     await sessionAPI.saveCopy(sessionId, result);
   }
   ```

4. **`StrategyStep.tsx`** — 同理，在 `regenerateCopy` 轮询完成后调用 `saveCopy`：
   ```typescript
   // 当前代码（L79-81）:
   const regenerate = await sessionAPI.regenerateCopy(sessionId, ALL_COPY_TARGETS);
   if (regenerate?.job_id) { await jobAPI.pollUntilDone(regenerate.job_id, ...); }
   
   // 需要新增：
   const copyResult = await sessionAPI.getCopy(sessionId);
   if (copyResult) {
     await sessionAPI.saveCopy(sessionId, copyResult);
   }
   ```

5. **`CopywritingStep.tsx`** — `forceRegenerateCopy` 回调中也需同样添加 `saveCopy` 调用。

**注意**：需确认后端 `GET /copy` 返回的数据结构是否直接可作为 `PUT /copy` 的请求体。如果 `GET` 返回的数据包含 `generated_fields` 字段，可能需要按后端要求提取 `generated_fields` 并构造 `PUT` 请求体。

### 任务 5：详情图编辑框文案来源改为 `copy_blocks`

**问题**：v2.1.0 文档明确要求"详情图编辑框文案来源用 `copy_blocks` 替代策略预览中的文案"。

**当前状态**：
- `DetailResultStep.tsx` 的 `DetailImage` 只有一个简单的 `text?: string` 字段
- 编辑时用户输入的自由文本通过 `regenerate` 的 instruction 传给后端
- 没有 `copy_blocks` 数据来源

**修改**（与任务 2 合并执行）：

1. 从 `GET detail-pages/results` 的 `panels[].copy_blocks` 获取真实渲染文案
2. 编辑框初始化值使用 `copy_blocks` 中的各字段（headline, supporting 等）
3. 编辑提交时将修改后的 `copy_blocks` 通过 `editAssetText` 发给后端
4. 这解决了"图片文字和编辑框不一致"的问题

## 实施顺序

1. **任务 1** → 类型扩展（基础设施，其他任务依赖）
2. **任务 4** → `saveCopy` 流程修复（独立，风险最低）
3. **任务 2 + 5** → 详情图编辑迁移（合并执行，改动最大）
4. **任务 3** → 主题方向确认（评估后决定是否需要改动）

## 风险与注意事项

1. **`saveCopy` 请求体结构**：需确认后端 `PUT /copy` 接受的请求体格式。如果 `GET /copy` 返回的数据不能直接传给 `PUT /copy`，需做数据转换。
2. **详情图编辑 UI 改动较大**：从单一文本输入变为结构化多字段编辑面板，需仔细设计 UI 交互和布局。
3. **`editAssetText` 对详情图的效果验证**：需确认后端 `editAssetText` 对 `detail_page` 类型 asset 的处理逻辑是否与 `main_gallery` 一致。
4. **向后兼容**：`copy_blocks` 是新增字段，需考虑后端未返回 `copy_blocks` 时的 fallback 处理（`null` 或默认空值）。
